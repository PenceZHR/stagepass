## 1. 要解决什么

两件事，都来自同一个观察：**流水线能拦住的东西太少，能交付的东西也太少。**

- 出厂 81 条 rubric 标准**全部 `blocking: false`**，也就是它们今天什么都不限制。
- `Done` 在代码里只是 `pipeline-ui-model.ts:209` 的一个 UI 标签——不是阶段、不在
  `RUBRIC_PHASES` 里、没有 prompt 模板。Retro 跑完把 change 置成 `DONE` 就结束了，
  **全仓没有任何东西产出「这个项目怎么用」**。

用户原话：「done 阶段就是用来给你交付的」。

## 2. 三级 rubric

### 2.1 三级的定义

| | 一级 · 绝对对齐 | 二级 · 阶段特有 | 三级 · 用户自加 |
|---|---|---|---|
| 判定方式 | **确定性代码检查 + AI 判定，两种都有** | AI 判定 | AI 判定 |
| 文本 | **不可删改** | 出厂给，可改 | 用户写 |
| 阻断 | **恒阻断** | 用户逐条勾 | 用户逐条勾 |
| 出口 | **无** | 撤下 criterion | 撤下 criterion |
| 在文件里 | 只读呈现 | 可编辑 | 可编辑 |

一级的内容按用户定义：**「整个项目写文件都有最基础的、绝对不能僭越的几条准则」**，
例如「严格对齐 PRD」「代码怎么写」「不许在计划范围外写文件」。其中：

- **确定性那部分已经有实现**，只是没在 rubric 面板露出：`stage-guard-service` 的
  `DEFAULT_STAGE_SCOPES` + runner 里的 `validatePlannedChanges`、`build-gate-service`
  的 `blockedGlobs`、`scope-check-service`（QA 期）。一级要做的是**把它们只读呈现
  出来**，让用户看得见「不能碰的是哪几条」，并保证它们关不掉。
- **AI 判定那部分**是新写的少数几条，措辞要极稳。

### 2.2 已知接受的风险：一级 AI 条款无出口

一级不可撤销，而现有设计里**唯一的出口就是撤下 criterion 本身**（§4.3.1）。
所以：**一级 AI 条款被判 `no`，没有任何出口。**

误判会发生，这是实证不是假设：出厂 critic 标准曾写「红方声称修复的每一个 gap」，
而三份 Spec 模板都定义「红方只指人类用户本人」；人类不产出修复声明，rubric 又规定
「无法确定就是 `no`」——**一条实际做到了的标准会被判 `no`**（已修，`8d59eb8`）。

用户明确选择「死锁就是正确行为」，主控提出异议两次并给出上述证据后判断未变。
**本设计按此执行，并记录为已知接受的风险。** 缓解只有两条，都不是出口：

1. 一级条款数量少、措辞稳，且每条上线前必须能举出「什么产物会让它答 `no`」的具体例子；
2. 已知会产生假 `no` 的缺陷类用测试钉死（角色词汇那条已做，见 `rubric-rollout.test.ts`）。

### 2.3 文件与 DB 的权责（approach C）

用户选择「落成项目里的可改文件」。rubric 不能照抄 `.ship/prompts/` 的做法——prompt
没有 DB 副本，而 rubric 的 DB 里带着版本钉扎、项目级/change 级作用域、以及
`criterionKey` 身份（已盖章的阻断项按 `RUBRIC:<criterionKey>` 持久化）。

**分工：文件管文本，DB 管状态。**

- 文件（`.ship/rubrics/<phase>-<role>.md`）：criterion 的 key、正文、层级、blocking 标记
- DB：版本、每轮判定、已盖章的阻断项

**同步规则：** 每次跑阶段对文件内容取哈希，与当前版本不同就**铸一个新版本，绝不
改旧版本**。已判过的轮次仍钉在它当时那一版——这是 §4.4「编辑 rubric 不使任何已盖章
gate 失效」的直接推论。

**删除即撤下。** 从文件里删掉一行 = 撤下该 criterion，正是既有的唯一出口，不新发明
机制。一级的行以只读形式写在文件里；删掉它们不生效（下次跑阶段会照原样补回）。

**前置已完成**：`criterionKey` 已改为显式书写（`282fcde`）。此前它由数组下标派生，
用户在文件中间插一行就会把后面每条静默改名、废掉已盖章的阻断项。

### 2.4 出厂标准如何到达存量项目（未决）

`ensureFactoryRubrics` **只播种缺失的 scope，已存在的永不更新**。所以 `8d59eb8` 修好的
措辞，PRJ-001 / PRJ-002 库里仍是旧的。

三级设计必须回答这个问题，但**本文暂不定方案**。候选：出厂标准带版本号，
新版本以「未生效的建议」形态出现在文件里由用户合入；或播种时按 key 比对正文，
仅在用户未改过该条时静默更新。**需要单独讨论后再定。**

### 2.5 阶段覆盖缺口

`RUBRIC_PHASES` 声明 11 个阶段，出厂 rubric 只覆盖 8 个：

- **Refine / QA / Merge 一条都没有**
- **Fix 声明了反方（`RUBRIC_PHASES_WITH_CRITIC`）却没有出厂标准**
- verdict 角色只有 Spec 有

`rubric-assessment.ts:43-52` 解释了 QA/Merge 不给反方的理由（确定性检查），但没解释
正方为何也空着。补齐属于本设计范围。

## 3. Done 交付阶段

### 3.1 形态

`Done` 从 UI 标签变成**真实执行阶段**，产出一份交付单。

- **粒度：每次 change 一份**（用户选择）。不可变、互不覆盖、天然可审计。
  代价是跑到第 10 个 change 时要翻 10 份才知道全貌——用户接受。
- 因为要回答「怎么引用」，必须读代码理解入口，所以是**带 provider 调用的真实阶段**，
  不是从既有产物拼装。

### 3.2 交付单必须回答的四件事

用户四项全选：

1. **怎么跑起来** —— 入口文件在哪、要不要装依赖、打什么命令或直接打开哪个文件、
   跑起来后应该看到什么。对《三军战线》就是「打开哪个文件能玩」。
2. **本次改动带来了什么** —— 新增修改了哪些能力、用户能看到的变化、怎么验证它生效了。
3. **文件地图** —— 新增修改的文件各自是干什么的，哪些是入口、哪些是内部实现，
   下次要改从哪下手。
4. **已知限制与没做的事** —— 明确不做的、被豁免的 P1、还没关闭的 gap、踩到的已知坑。

第 4 项有一个确定性来源：被豁免的 P1、未关闭的 gap、`requirement_gaps` 里的 open 行
都在库里。**这部分应由代码填，不由模型自述**——模型自述「我没做什么」正是它最不可靠
的方向。

### 3.3 落点

`.ship/changes/<changeId>/delivery.md`，与该 change 的其它产物同处，并注册为该阶段的
正式产物（`phase-artifact-service`）。

### 3.4 Done 的 rubric

Done 进 `RUBRIC_PHASES`，producer 角色。一级候选条款：

- 「交付单里写的启动方式，我确认在当前仓库状态下真的能跑通」——这条是本阶段存在的
  理由，若它可以答 `yes` 而实际跑不起来，整个交付就是假的。
- 「已知限制一节列出的内容，与库里 open 的 gap / 已豁免的 P1 一致」——可与确定性来源
  交叉核对。

## 4. 实施顺序

1. Done 阶段（独立、不动既有行为、能立刻交付价值）
2. 一级确定性条款的只读呈现（把已有检查接进面板，不新增拦截）
3. 一级 AI 条款 + 恒阻断（**第一次真的会拦人，需要在废项目上先验**）
4. `.ship/rubrics/` 文件读写 + 版本铸造
5. 补齐 Refine / QA / Merge / Fix-critic 的出厂标准
6. §2.4 出厂标准更新传播（需先单独讨论）

第 3 步是第一个不可逆的点：一级恒阻断上线后，任何一次 AI 误判都会卡死当时在跑的
change。上线前必须在 PRJ-002/003 这类废项目上跑通完整一轮。
